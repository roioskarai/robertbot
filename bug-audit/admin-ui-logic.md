> סבב סריקה: **איתור ותיעוד בלבד — ללא תיקון קוד.**
> תחום: `app/src/app/admin/**`, `app/src/app/dashboard/**`, ורכיבי `app/src/components/**` שבהם הם משתמשים.
> תאריך: 2026-06-14

## הערה מקדימה על "באג ה-FAQ" שדווח

הבעלים דיווח: *"במסך שאלות-תשובות, לחיצה על שאלה לא פותחת את התשובה — ובמקום זה השאלה נמחקת."*

חשוב להבין: בקוד יש **שני** מסכי FAQ שונים לחלוטין:

1. **FAQ אקורדיון (קריאה בלבד)** — בדף הנחיתה (`app/src/app/page.tsx`). שם הלחיצה *כן* פותחת/סוגרת תשובה. הקוד שם **תקין**.
2. **עורך FAQ (עריכה)** — בעורך הבוט בדשבורד (`app/src/app/dashboard/page.tsx`) ובאונבורדינג. **כאן** מתרחש הבלבול שהבעלים תיאר: זה לא אקורדיון בכלל, אלא שורה עם שני שדות קלט שקופים + כפתור מחיקה `×`. ה-UI הזה גורם למשתמש לחשוב שזו "שאלה ללחיצה" שתיפתח — אבל הלחיצה במקום הלא-נכון מפעילה מחיקה. הסבר מלא ב-**באג #1**.

הדוח ממוקד בעורך הדשבורד כי שם הבעלים עובד, ושם ההתנהגות תואמת בדיוק את התלונה.

### 🔴 #1 — עורך ה-FAQ בדשבורד: השדות בלתי-נראים והלחיצה "נופלת" על כפתור המחיקה
**קובץ:** `app/src/app/dashboard/page.tsx:540-548`
**CSS תומך:** `app/src/app/dashboard/dashboard.module.css:318-321`

**הלוגיקה השגויה / מקור הבלבול:**
מבנה כל שורת FAQ הוא:
```tsx
<div className={c("faq-row")} key={i}>
  <input value={f.question} placeholder="שאלה" onChange={...} />   // שדה שקוף
  <input value={f.answer}   placeholder="תשובה" onChange={...} />   // שדה שקוף
  <span className={c("faq-del")} onClick={...filter(j !== i)}>×</span>  // מחיקה
</div>
```
ה-CSS הופך את שני ה-`<input>` ל**בלתי-נראים** כשדות: `border:none; background:transparent` (שורה 319). כלומר אין כל סימן ויזואלי ששאלה ותשובה הם שני שדות עריכה נפרדים — הם נראים כמו טקסט פשוט בשורה. משתמש (במיוחד לא-טכני) רואה "שאלה" ומנסה ללחוץ עליה כדי "לפתוח אותה". בפועל:
- אם הוא לוחץ על אזור הטקסט → לא קורה כלום נראה לעין (סתם מיקוד בשדה ריק/שקוף) → נראה "תקוע".
- אם הוא לוחץ קצת ימינה, על ה-`×` (שגם הוא דהוי, `color:var(--t4)`, שורה 320) → **השאלה נמחקת**.

**התנהגות נצפית מול מצופה:**
- נצפה: לחיצה על שאלה → השורה נעלמת (נמחקת), או "כלום לא קורה".
- מצופה: לחיצה פותחת/מציגה את התשובה לעריכה ברורה, וכפתור מחיקה נפרד וברור.

**תיקון מוצע:** לעצב את ה-`<input>` כשדות נראים (גבול/רקע), להפוך את ה-`×` לכפתור מחיקה גדול וברור עם `title`/`aria-label="מחק שאלה"`, ולשקול אישור (`confirm`) לפני מחיקה — כך שלחיצה בטעות לא תמחק תוכן.

### 🔴 #2 — מחיקת בוט ללא אישור: לחיצה אחת מוחקת בוט שלם מיד
**קובץ:** `app/src/app/dashboard/page.tsx:238-246` (handler) · `581` (כפתור)

**הלוגיקה השגויה:**
```tsx
<button className={c("btn btn-danger btn-sm")} onClick={deleteBot}>מחק בוט</button>
```
`deleteBot()` שולח `DELETE` מיידית ללא כל `confirm()`. הכפתור "מחק בוט" יושב בשורה אחת ליד "ביטול" ו"שמור שינויים" (שורות 581-583), כך שלחיצה שגויה הורסת בוט שלם (כולל שיחות/חיבור וואטסאפ) ללא דרך חזרה ובלי אזהרה. שים לב: בדף ה-preview כן יש `confirm` לפעולה הרבה פחות הרסנית (הפעלת בוט), אבל למחיקה — אין.

**התנהגות נצפית מול מצופה:**
- נצפה: לחיצה בודדת → הבוט נמחק מיד, toast "הבוט נמחק".
- מצופה: דיאלוג אישור ("למחוק את הבוט X? פעולה בלתי-הפיכה") לפני המחיקה.

**תיקון מוצע:** לעטוף את `deleteBot` ב-`confirm(...)` (או מודל אישור), ולהרחיק ויזואלית את כפתור המחיקה משאר הכפתורים.

### 🟠 #3 — חיבור מספר וואטסאפ ידני לא מפעיל את הבוט, אך הניתוק כן מכבה אותו (לוגיקה א-סימטרית)
**קובץ:** `app/src/app/dashboard/page.tsx:574` (חיבור ידני) מול `560` (ניתוק)

**הלוגיקה השגויה:**
- **ניתוק** מספר: `onClick={() => setEditBot({ ...editBot, whatsapp_number: null, active: false })}` — מכבה את הבוט (`active:false`). הגיוני.
- **חיבור ידני**: ה-`<input>` רק מעדכן `whatsapp_number` (`onChange={(e) => setEditBot({ ...editBot, whatsapp_number: e.target.value })}`) ו**אינו** מדליק `active`. כפתור "חבר" (שורה 575) רק מציג toast "נשלח קוד אימות" ולא משנה state כלל.
- לעומת זאת מסלול ה-**חיבור האוטומטי** (`ConnectWhatsApp`, שורה 567) כן עושה `active: true`.

כך נוצר חוסר-עקביות: הבוט מתחבר ידנית למספר אך נשאר כבוי, והמשתמש לא מבין למה הבוט "מחובר" אבל לא עונה. בנוסף, כל הקלדה בשדה הידני (תו אחר תו) דורסת את `whatsapp_number` — אם הקלדה לא הושלמה, נשמר מספר חלקי לא תקין.

**התנהגות נצפית מול מצופה:**
- נצפה: חיבור ידני → מספר נשמר אך `active` נשאר false; "חבר" לא עושה דבר אמיתי.
- מצופה: חיבור מוצלח → הבוט מודלק, או לפחות התנהגות סימטרית לניתוק.

**תיקון מוצע:** כפתור "חבר" צריך לבצע ולידציה של המספר ולהדליק `active:true` (כמו במסלול האוטומטי), ולא רק להציג toast.

### 🟡 #6 — `returnToBot` מסיר את השיחה מ-`convs` גם בכשל שרת (optimistic ללא rollback)
**קובץ:** `app/src/app/dashboard/page.tsx:948-955`

**הלוגיקה השגויה:**
```tsx
await fetch(`/api/conversations/${activeConvId}/return`, { method: "POST" }).catch(() => {});
setConvs((cs) => cs.filter((cv) => cv.id !== activeConvId));
```
ה-`catch(() => {})` בולע שגיאות רשת, ובכל מקרה (גם אם השרת החזיר 500 או הבקשה נכשלה) ה-`setConvs(...filter)` מסיר את השיחה מהרשימה ומציג "השיחה הוחזרה ל-Robert". כלומר: השיחה נעלמת מה-Inbox של הנציג אך בפועל לא הוחזרה בשרת — היא "תקועה" במצב human ללא נציג שרואה אותה. אין rollback בעת כשל. (`sendReply` מתנהג דומה — תמיד toast "נשלחה" גם אם ה-fetch נכשל, שורות 939-945.)

**התנהגות נצפית מול מצופה:**
- נצפה: כשל שרת → השיחה נעלמת מה-UI ו-toast הצלחה, אך לא הוחזרה בפועל.
- מצופה: להסיר מה-UI רק אחרי `res.ok`, ולהחזיר/להציג שגיאה בכשל.

**תיקון מוצע:** לבדוק `res.ok` לפני `setConvs(filter)` ולפני toast ההצלחה; בכשל — toast שגיאה ולהשאיר את השיחה.

### 🟡 #8 — שדות "הגדרות חשבון" הם uncontrolled עם `defaultValue`; "שמור" לא שומר כלום

- מצופה: שמירה אמיתית לשרת, או לפחות אי-הצגת "נשמר" כשלא נשמר.

**תיקון מוצע:** לחבר את השדות ל-state נשלט + קריאת API לשמירה (כמו ב-`saveBot`), או לסמן מפורשות שהמסך עדיין דמו.

### ⚪ #14 — אדמין/בוטים: `useEffect(load, [])` מעביר את ה-event של React כארגומנט ל-`load`
**קובץ:** `app/src/app/admin/(panel)/bots/page.tsx:32` · גם `billing/page.tsx:21` · `security/page.tsx:21`

**הלוגיקה השגויה:**
`useEffect(load, [])` ו-`useEffect(loadMe, [])` — מעבירים את הפונקציה ישירות ל-`useEffect`. זה עובד כי React קורא לה ללא ארגומנטים, אבל זה דפוס שביר: אם `load` תקבל בעתיד פרמטר (כמו `load(e)` מ-onClick שכבר קיים בשורה 46/32), קל לטעות. הבעיה האמיתית-יותר: אותה `load` משמשת גם כ-`onClick={load}` (bots:46, billing:32) — שם React מעביר את אובייקט ה-event כארגומנט. כרגע `load` מתעלמת מארגומנטים אז אין נזק, אבל זה דפוס סיכון. (חומרה נמוכה — לא משבש כרגע.)

**התנהגות נצפית מול מצופה:** ללא תקלה נצפית כיום; סיכון תחזוקתי.
**תיקון מוצע:** לעטוף: `useEffect(() => { load(); }, [])` ולוודא ש-handlers שמקבלים event לא מועברים ישירות.

### ⚪ #15 — אדמין/משתמשים: עדכוני `patch` ללא בדיקת תשובה; כשל שרת מוצג כהצלחה
**קובץ:** `app/src/app/admin/(panel)/users/page.tsx:57-64` (`patch`)

**הלוגיקה השגויה:**
```tsx
async function patch(id, body, msg?) {
  await fetch(`/api/admin/users/${id}`, { method:"PATCH", ... });
  load();
  if (msg) showToast(msg);  // מוצג תמיד, גם אם ה-PATCH נכשל
}
```
אין בדיקת `res.ok`. אם עדכון מסלול/סטטוס/חסימה נכשל בשרת, עדיין מוצג toast הצלחה ("מסלול עודכן" / "משתמש חסום"). `load()` אמנם ימשוך מחדש את הערך האמיתי (וה-select יחזור), אבל ה-toast המטעה כבר נראה, והמנהל יחשוב שהפעולה הצליחה. (פעולות אדמין רגישות — מסלול, חסימת משתמש.)

**התנהגות נצפית מול מצופה:**
- נצפה: PATCH נכשל → toast "עודכן" אך הערך חוזר לקדמותו אחרי `load`.
- מצופה: לבדוק `res.ok` ולהציג שגיאה בכשל.

**תיקון מוצע:** לבדוק `if (!res.ok) { showToast("העדכון נכשל", false); return; }` לפני `load()` וה-toast החיובי.

### ⚪ #16 — אדמין/בוטים: `setLoading` ב-`load` ללא `.catch` → "טוען" לנצח בכשל רשת
**קובץ:** `app/src/app/admin/(panel)/bots/page.tsx:28-31` · גם `billing/page.tsx:17-20`

**הלוגיקה השגויה:**
```tsx
function load() {
  setLoading(true);
  fetch("/api/admin/bots").then(r=>r.json()).then(d=>{setBots(...); setLoading(false);});
}
```
אין `.catch`. אם ה-`fetch` נכשל (רשת/500) או `r.json()` זורק, `setLoading(false)` לעולם לא ירוץ והמסך נשאר ב-skeleton "טוען" לצמיתות, ללא הודעת שגיאה. (`users/page.tsx` ו-overview עדיפים — overview יש לו `.catch`; users משתמש ב-async/await אך גם שם אין try/catch סביב ה-fetch.)

**התנהגות נצפית מול מצופה:**
- נצפה: כשל טעינה → skeleton אינסופי.
- מצופה: מצב שגיאה + אפשרות רענון.

**תיקון מוצע:** להוסיף `.catch(() => setLoading(false))` + מצב שגיאה לכל ה-loaders של האדמין.
